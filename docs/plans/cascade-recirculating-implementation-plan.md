# PDF三级循环水箱场景实施计划

## 决策

新增独立场景 `cascade_recirculating`，保留现有开式三加热器场景
`cascade`。二者属于同一水箱模型族，但不共享场景标识、动作空间、参数档案
或任务目录。

设计依据为仓库上级目录中的《三级水箱加热系统.pdf》。该文件是方案讨论图，
不是竣工图或设备标定记录；只有闭式循环拓扑、P101/V12/V23设备关系、Tank 1
单台2 kW加热器和硬件联锁架构可视为设计来源。

## 阶段1：场景契约与可运行基础（已完成）

- 增加6状态、4动作的 `RecirculatingCascadeModel`。
- 实现无正常补水/产品流的名义闭式循环质量与能量方程。
- 注册 `cascade_recirculating`，增加参数档案和显式默认任务。
- 将2 kW加热器标为 `design-specified`，其余缺失设备参数明确标为临时值。
- 验收：模型契约、注册、模型卡、有限导数、单步积分和闭式总质量残差通过。

## 阶段2：动力学独立验证（已完成）

- 独立重算三个罐的质量和能量残差。
- 验证P101从Tank 3移除的质量和焓与Tank 1入口完全一致。
- 增加numeric/CasADi导数、动作限幅和积分轨迹一致性测试。
- 检查默认工作点和步长收敛性，重新选择仅用于基准的初始动作。

## 阶段3：装置安全支路（已完成）

- 增加Tank 1和Tank 2到Tank 3的无源溢流回路，而不是把所有高液位只当作
  无质量去向的终止事件。
- 完成P101低液位停泵、H1干烧和超温联锁的诊断与实际能耗口径。
- 区分硬件联锁动作、软件安全事件和真正需要终止仿真的边界。
- 所有未知阈值继续标为 `assumed-benchmark`，不得解释为设备保护定值。

## 阶段4：任务与控制器（已完成）

- 增加装置调试、温度跟踪、扰动抑制和安全验证任务。
- 只定义循环流量或热梯度目标，不沿用开式 `cascade` 的产品流量经济目标。
- 为PID、MPC和Oracle增加与4动作结构一致的基线配置与可执行验证。
- 验证默认温度目标满足 `T1 >= T2 >= T3` 的单热源物理方向。
- 在模型与任务两层显式拒绝 `economic` 目标，避免零价值/零成本配置被误解为
  有效生产经济学。
- 验收：5个任务档案通过契约校验，扰动按时作用并向控制器与诊断接口公开，
  PID/MPC/Oracle均完成4动作短回合运行且无约束违规。

## 阶段5：证据与交付（已完成）

- 生成模型卡、场景文档和独立准确性/真实性报告。
- 建立PDF逐项来源矩阵，记录文档内部总功率口径不一致问题。
- 运行专项测试和完整回归测试，确保原 `cascade` 的结果与接口不变。
- 将现场仍需采集的罐体尺寸、V12/V23流量试验、P101曲线、冷却曲线和联锁
  定值列为设备验证清单。
- 配套输出机器可读验证JSON和7页A4中文版PDF；PDF逐页渲染检查无乱码、裁切或
  表格重叠。
- 最终验收：模型卡生成一致性检查、JSON语法、`git diff --check`和完整仓库
  `197 passed, 24 subtests passed`全部通过。

## 不在当前证据范围内的声明

- 不从参考布置图比例反推水箱有效面积或容积。
- 不把约0.30 m平台落差直接换算为V12/V23流量系数。
- 不把PDF中的泵/电子设备合计功率当作P101铭牌功率。
- 在取得竣工资料和实验数据前，不声明该场景是经过验证的数字孪生。
