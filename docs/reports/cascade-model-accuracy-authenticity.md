## 1. 结论摘要

当前模型是一个结构明确、守恒方程实现一致、边界行为可执行、可重复的控制算法基准模型。它已经修复输入校验、空罐排流、干烧、高温联锁、边界终止、后端输出、参数元数据和连续生产经济口径等软件缺陷。

它仍然不是经过真实装置数据验证的数字孪生。仓库没有设备图纸、泵阀曲线、加热器标定、参数辨识数据或独立验证时间序列，因此不能给出真实设备 RMSE、MAE、R² 或“准确率百分比”。

## 2. 模型契约

### 2.1 拓扑

```text
冷水 -> 进料泵 -> Tank 0 -> 阀 0 -> Tank 1 -> 阀 1 -> Tank 2 -> 阀 2 + extra_outflow
                       加热器 0          加热器 1          加热器 2
```

每个出口是单向自由排水，不使用下游背压，也不允许反向流动。

### 2.2 状态、动作与输出顺序

状态保持液位/温度交错：

\[
x=[h_0,T_0,h_1,T_1,h_2,T_2]
\]

动作保持泵、三个阀、三个加热器：

\[
u=[u_p,u_{v0},u_{v1},u_{v2},u_{H0},u_{H1},u_{H2}],\quad 0\le u_j\le1
\]

受控输出为重排后的原始物理状态：

\[
y=[h_0,h_1,h_2,T_0,T_1,T_2]
\]

`controlled_output()` 不裁剪液位；仅 UI 使用的 `display_outputs()` 会把数值后端的负液位显示为 0。控制器、KPI、约束与 CasADi 均读取原始状态。

### 2.3 主要参数

| 参数 | 当前值 | 状态 |
| --- | ---: | --- |
| 单罐横截面积 | 0.15 m² | `legacy-unverified` |
| 最大液位 | 0.8 m | `legacy-unverified` |
| 出口系数 | 0.0026 m²·⁵/s | `legacy-unverified` |
| 单罐散热 UA | 40 W/K | `legacy-unverified` |
| 单加热器最大电功率 | 90 kW | `legacy-unverified` |
| 最大泵流量/功率 | 0.0016 m³/s / 1.5 kW | `legacy-unverified` |
| 冷水/环境温度 | 15/20 °C | `legacy-unverified` |
| 加热器最低液位 | 0.05 m | `assumed-benchmark` |
| 加热器温度跳车 | 92 °C | `assumed-benchmark` |
| 温度硬终止 | 120 °C | `assumed-benchmark` |

`heater_efficiency` 现在表示“电功率传递给液体的比例”，基准扰动范围为 `[0.4,1.0]`。超过 1 的容量变化必须通过 `heater_max` plant regime 表达，不能再解释为效率。

## 3. 方程与物理含义

进料泵流量：

\[
q_p=u_p q_{p,\max} f_p
\]

第 \(i\) 个自由排水阀：

\[
q_i=C_vu_{vi}\sqrt{\max(h_i,0)}
\]

当对应液位小于等于 0 时，阀流量严格为 0；第三罐为空时，`extra_outflow` 也被关闭。

质量衡算：

\[
\dot h_0=\frac{q_p-q_0}{A},\quad
\dot h_1=\frac{q_0-q_1}{A},\quad
\dot h_2=\frac{q_1-q_2-q_{extra}}{A}
\]

温度衡算：

\[
\dot T_i=
\frac{q_{in,i}(T_{in,i}-T_i)}{A\max(h_i,h_{floor})}
+\frac{P_{liq,i}-UAf_L(T_i-T_{amb})}
{\rho c_pA\max(h_i,h_{floor})}
\]

液体获得的加热功率为：

\[
P_{liq,i}=u_{Hi}P_{H,\max}\eta_H I_i
\]

其中联锁 \(I_i\) 只有在 \(h_i\ge0.05\,m\) 且 \(T_i<92\,°C\) 时为 1，否则为 0。

## 4. 边界与安全语义

模型不通过无质量去向的状态投影“修复”越界：

- `h_i < 0`：环境硬终止，原因为 `negative_level`；
- `h_i > height_max`：环境硬终止，原因为 `overflow`；
- `T_i >= temperature_hard_limit`：环境硬终止，原因为 `temperature_hard_limit`；
- `h_i < heater_min_level`：仅关闭对应加热器；
- `T_i >= temperature_trip`：仅关闭对应加热器。

前三类原因不依赖 `terminate_on_runaway`，并通过 `termination_reason` 与 `safety_events` 报告。由于使用固定步长积分，终止状态可能略微跨过边界；系统不会在该状态上继续下一个环境转换。

审计复现：

| 工况 | 结果 |
| --- | --- |
| 0.799 m 首罐、泵全开、0.5 s | 终止；`overflow`；首罐 0.804333 m |
| 0.001 m 第三罐、额外排水 0.002 m³/s、0.5 s | 终止；`negative_level`；第三罐约 -2.22e-5 m |
| 三罐液位 0.04 m、加热全开 | 三罐传液功率均为 0 W |
| 三罐温度 92 °C、加热全开 | 三罐传液功率均为 0 W |

这些结果验证软件联锁和终止口径，不代表 0.05、92 或 120 的设定来自安全认证。

## 5. 批式与连续生产语义

运行模式属于任务，而不是物理参数：

- 直接环境与 `legacy-default`：`mode=batch`，允许零吞吐量；
- `continuous-benchmark`：`mode=continuous`，`product_flow_sp=min_product_flow=4.0e-4 m³/s`。

环境模型和 runner 创建的控制器模型通过同一路径读取任务 operation。经济 Oracle 使用同一个模型配置。连续模式经济收益包括产品流量、电功率、温度/液位违反和归一化流量短缺。

默认设定值下的连续稳态需求：

\[
u^*=[0.25,0.2293403,0.2293403,0.2293403,0.3787556,0.2924,0.2990667]
\]

| 项目 | 数值 |
| --- | ---: |
| 产品流量 | 0.0004 m³/s |
| 泵电功率 | 0.375 kW |
| 三台加热器总电功率 | 87.320 kW |
| 总理想电功率 | 87.695 kW |
| 效率降到 0.5 时总理想功率 | 175.015 kW |
| 关泵关阀的单位时间经济收益 | -100.0 |
| 按目标稳态生产的单位时间经济收益 | -21.3865 |

因此，在声明的连续基准中，零泵零阀策略不会仅因低能耗成为最优。上述收益权重与吞吐目标仍是基准设定，不是实际价格或额定产能。

## 6. 剩余建模假设

- 水不可压缩，密度和比热为常数。
- 三个罐面积、出口系数、UA 和加热器容量相同。
- 每个罐瞬时完全混合，无温度分层、死区或短路流。
- 管路无体积、传输延迟、压降动态和散热。
- 自由排水不考虑下游背压、虹吸或反向流动。
- 泵流量/功率与命令线性，不使用扬程和效率曲线。
- 阀门无死区、回差、压差特性和动作动态。
- 加热器、罐壁和传感器无独立热容、滤波或延迟。
- 不模拟蒸发、沸腾、气泡、气蚀、溢流流量和相变。
- `h_floor` 仅避免温度方程除零，不是残余液体模型。

这些假设限制快速瞬态、极低液位、高温和真实执行器非线性的预测能力。

## 7. 内部验证结果

审计环境：Python 3.14.6；随机种子 `20260717`。

### 7.1 通用就绪检查

`validate_model_readiness("cascade")` 通过模型契约、有限导数、有限输出、RK4 配置和有限积分步。该检查仍明确把质量衡算、能量衡算和参考参数真实性列为未检查，因此不能解释为真实装置验证。

### 7.2 守恒残差

在 200 个内部工作点（液位 0.06–0.75 m、温度 15–85 °C、动作 0–1）上独立重算：

| 检查 | 最大绝对残差 |
| --- | ---: |
| 系统总质量衡算 | 3.253e-19 m³/s |
| 单罐能量衡算 | 1.746e-10 W |

残差处于浮点舍入量级，说明代码在已测内部域忠实实现声明方程。

### 7.3 连续目标稳态

- 所有 7 个动作均位于 `[0,1]`；
- 最大绝对状态导数：1.388e-17（对应状态单位/s）；
- 从该点积分 300 s，最大状态漂移：0。

### 7.4 数值步长收敛

这只支持已测平滑轨迹，不能自动推广到未来的刚性动力学或联锁切换瞬间。

## 10. 迁移说明

1. 以前允许 `heater_efficiency > 1` 的调用现在会失败；容量变化应使用 `heater_max` plant regime。
2. `controlled_output()` 现在返回原始液位；仅展示层需要非负格式时使用 `display_outputs()`。
3. 负液位、溢流和温度硬上限现在无条件终止；调用方必须处理 `terminated`、`termination_reason` 和 `safety_events`。
4. 实际加热能耗按联锁后的电功率记录；被联锁加热器不再计入实际 plant 能耗。
5. 连续生产经济比较必须显式选择 `task="continuous-benchmark"`；批式结果不能解释为连续生产排名。
6. `ideal_energy_kw()` 现在使用任务吞吐目标并包含泵功率和效率修正，不再由当前泵动作决定。

## 11. 真实装置验证所需证据

至少需要：

- 三个罐的面积—液位曲线、有效高度和溢流结构；
- 泵命令—流量—扬程—功率曲线；
- 阀开度—流量—压差曲线及回差；
- 加热器电功率、传热效率、浸没要求和认证跳车定值；
- 罐壁/保温材料、热容和自然冷却数据；
- 液位、温度、流量和功率传感器校准记录；
- 带统一时间戳的状态、动作、扰动、流量和功率动态数据；
- 与辨识集隔离的验证工况。

验证集至少报告每个输出的 RMSE、MAE、偏差、NRMSE，30/60/300 s 自由运行误差，稳态增益、时间常数、纯滞后误差，残差自相关，参数置信区间和工况分区结果。

## 12. 给 ChatGPT 或其他模型的读取规则

1. 把配套 JSON 视为数值结果的机器可读来源。
2. `readiness.passed=true` 仅表示软件/数值就绪。
3. 不得声称模型已由真实装置验证，也不得编造准确率、RMSE 或额定产能。
4. 区分“内部已验证”“假设基准”“真实装置未验证”。
5. 不得声称液位会被钳位；当前语义是保留越界状态并硬终止。
6. 不得把 0.05、92、120 或 4.0e-4 当成设备认证值。
7. 不得把 `h_floor` 当成真实残余液位或空罐热模型。
8. 连续经济结论只适用于 `continuous-benchmark`；默认/legacy 是批式。
